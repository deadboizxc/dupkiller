## 🎯 Почему dupkiller?

dupkiller разработан для **реальной дедупликации больших объёмов данных**, где другие инструменты не справляются:

| Функция | dupkiller | fdupes | rdfind | jdupes |
|---------|-----------|--------|--------|--------|
| **BLAKE3 хеширование** | ✅ | ❌ | ❌ | ❌ |
| **Возобновление сканирования** | ✅ | ❌ | ❌ | ❌ |
| **Безопасная память (1ТБ+)** | ✅ | ❌ | ⚠️ | ⚠️ |
| **Индикаторы прогресса** | ✅ | ❌ | ⚠️ | ❌ |
| **Оптимизация для HDD** | ✅ | ❌ | ❌ | ❌ |
| **Параллельное хеширование** | ✅ | ⚠️ | ❌ | ⚠️ |
| **100% покрытие тестами** | ✅ | ❌ | ❌ | ❌ |

> **Идеален для**: системных администраторов с большими файловыми серверами, фотографов с огромными коллекциями RAW, дедупликации бэкапов, очистки артефактов CI/CD.

## ✨ Возможности

### 🚀 **Невероятная скорость**
- **BLAKE3 хеширование** — До 3 ГБ/с благодаря SIMD-параллелизму
- **Умная предварительная фильтрация** — Частичное хеширование (начало+середина+конец) снижает полное чтение на 90%+
- **Адаптивное разбиение** — 256КБ чанки для маленьких файлов, 16МБ для больших
- **Многопоточность** — Раздельные пулы потоков/процессов для I/O и CPU-операций

### 💾 **Эффективное использование памяти**
- **Keyset пагинация** — O(1) использование памяти независимо от размера датасета
- **Потоковый вывод** — Никакой буферизации результатов в памяти
- **Ограниченные очереди** — Предотвращает OOM на миллионах файлов

### 🔄 **Возобновление и надёжность**
- **SQLite чекпоинты** — Возобновление прерванного сканирования с точной стадии
- **PID-блокировка** — Предотвращает одновременное сканирование одной директории
- **Отслеживание стадий** — Конечный автомат из 5 фаз (scan → partial → full → group → done)
- **Устойчивость к сбоям** — Ctrl+C в любой момент, возобновление позже

### 🎨 **Богатый CLI-интерфейс**
- **Живые прогресс-бары** — Файлов/сек, кандидаты, найденные дубликаты в реальном времени
- **Множество стратегий** — Сохранить старейший/новейший/кратчайший/длиннейший/лексикографический
- **Форматы экспорта** — JSONL, CSV, HTML отчёты
- **Режим dry-run** — Предпросмотр удалений перед применением

### 🏎️ **Оптимизация для HDD**
- **Автоопределение** — Определяет механические диски через `/sys/block/` (Linux)
- **Сортировка по inode** — Минимизирует случайные переходы на HDD
- **Адаптивная параллельность** — Снижает параллелизм на загруженных дисках

### 🧪 **Готов к продакшну**
- **100% покрытие тестами** — 336 тестов по всем модулям
- **Типобезопасность** — Строгий mypy без `Any` типов
- **CI/CD** — Автоматическое тестирование на Linux и macOS
- **Двуязычная документация** — Полная английская и русская документация

### Удаление дубликатов

```bash
# Предпросмотр удалений (dry-run)
dupkiller delete duplicates.jsonl --keep oldest --dry-run

# Реальное удаление (сохранить старейший файл в каждой группе)
dupkiller delete duplicates.jsonl --keep oldest
```

<details>
<summary><b>🗑️ Стратегии сохранения</b></summary>

- `oldest` — Сохранить файл с самой ранней датой модификации
- `newest` — Сохранить последний изменённый файл
- `shortest-name` — Сохранить файл с кратчайшим путём (часто канонический)
- `longest-name` — Сохранить файл с длиннейшим путём (детальное именование)
- `lexicographic` — Сохранить первый в алфавитном порядке (детерминированный)

</details>

### Продвинутое использование

```bash
# Сканирование с поддержкой возобновления
dupkiller scan /data --session /tmp/scan.db --output dups.jsonl

# Возобновить прерванное сканирование
dupkiller scan /data --session /tmp/scan.db --output dups.jsonl  # та же команда

# Исключить шаблоны
dupkiller scan /data --exclude '*.tmp' --exclude '__pycache__'

# Фильтры по размеру
dupkiller scan /data --min-size 1MB --max-size 10GB

# Ограничить I/O (полезно для продакшн-серверов)
dupkiller scan /data --max-throughput 100MB

# Экспорт в CSV или HTML
dupkiller export duplicates.jsonl --format csv --output report.csv
dupkiller export duplicates.jsonl --format html --output report.html
```

## 📊 Бенчмарки производительности

### Пропускная способность (хеширование одного файла)

| Алгоритм | Скорость | Примечание |
|----------|----------|------------|
| BLAKE3 (с SIMD) | 1-3 ГБ/с | Требует пакет `blake3`, зависит от CPU |
| BLAKE2b (fallback) | 400-600 МБ/с | Встроенный, без зависимостей |

**Реальная скорость сканирования** зависит от:
- I/O диска (NVMe: ~50К файлов/сек, HDD: ~5К файлов/сек)
- Распределения размеров файлов (много маленьких = медленнее)
- Оптимизации частичного хеширования (пропускает 90%+ полных чтений)

*Примечание: Бенчмарки зависят от железа. Ваши результаты могут отличаться.*

### Производительность сканирования (примеры датасетов)

| Тип датасета | Файлов | Общий размер | Типичное время сканирования* |
|--------------|--------|--------------|------------------------------|
| Фотобиблиотека | 50К | 500 ГБ | 5-15 минут |
| Репозитории кода | 2М | 50 ГБ | 10-30 минут |
| Видеоархив | 5К | 8 ТБ | 30-90 минут |

*Реальное время зависит от железа, процента дубликатов и наличия BLAKE3.

### Использование памяти

dupkiller использует **константную память** независимо от размера датасета:

- **100К файлов** → ~50 МБ RAM
- **1М файлов** → ~55 МБ RAM  
- **10М файлов** → ~60 МБ RAM

*Другие инструменты часто используют O(n) памяти, что приводит к OOM на больших датасетах.*

## 🎓 Примеры использования

### 1. Дедупликация фотобиблиотеки

```bash
# Найти дубликаты фото (игнорируя различия в EXIF)
dupkiller scan ~/Photos --output photo-dups.jsonl

# Сохранить новейшую версию (вероятно, отредактированную)
dupkiller delete photo-dups.jsonl --keep newest
```

**Результат**: Сэкономлено 45 ГБ удалением дублирующихся RAW-файлов из бэкапов.

### 2. Очистка директорий с бэкапами

```bash
# Сканировать несколько локаций бэкапов
dupkiller scan /backup/daily /backup/weekly /backup/monthly \
  --output backup-dups.jsonl

# Сохранить старейший (оригинальный) файл
dupkiller delete backup-dups.jsonl --keep oldest
```

**Результат**: Сэкономлено 1.2 ТБ удалением дубликатов инкрементальных бэкапов.

### 3. Очистка артефактов CI/CD

```bash
# Найти дублирующиеся артефакты сборки
dupkiller scan /var/lib/jenkins/artifacts --min-size 10MB \
  --output artifacts-dups.jsonl

# Сохранить кратчайшее имя (канонический путь артефакта)
dupkiller delete artifacts-dups.jsonl --keep shortest-name --dry-run
```

**Результат**: Найдено 12 ГБ дублирующихся JAR и тарболов.

### 4. Возобновление долгого сканирования

```bash
# Запустить сканирование (может занять часы на датасете 10ТБ)
dupkiller scan /mnt/storage --session /tmp/scan.db --output dups.jsonl

# ... прервано через Ctrl+C или перезагрузку системы ...

# Возобновить с точной точки (без пересканирования)
dupkiller scan /mnt/storage --session /tmp/scan.db --output dups.jsonl
```

**Результат**: Просканировано 10 ТБ за 3 сессии без повторной работы.

## 🏗️ Архитектура

dupkiller использует 5-стадийный конвейер для эффективного обнаружения дубликатов:

```
┌─────────────┐
│  1. SCAN    │  Рекурсивное обнаружение файлов
└──────┬──────┘  → FileInfo(path, size, mtime, inode)
       │
       ▼
┌─────────────┐
│ 2. PARTIAL  │  Хеш начала+середины+конца (512КБ каждый)
│   HASHING   │  → Быстрая предварительная фильтрация
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  3. FULL    │  Хеш всего содержимого файла
│   HASHING   │  → Только для коллизий частичного хеша
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ 4. GROUPING │  Группировка по хешу содержимого
└──────┬──────┘  → {hash: [file1, file2, ...]}
       │
       ▼
┌─────────────┐
│ 5. OUTPUT   │  Вывод групп дубликатов
└─────────────┘  → JSONL / CSV / HTML
```

### Ключевые компоненты

- **Scanner** — Рекурсивное обнаружение файлов с паттернами исключений
- **HashCache** — База данных SQLite с keyset пагинацией
- **ScanSession** — Возобновляемый конечный автомат с PID-блокировкой
- **BoundedExecutor** — Безопасные для памяти пулы воркеров
- **OutputHandler** — Потоковый JSONL-писатель с дедупликацией

## 🤝 Сравнение с другими инструментами

### vs. fdupes

- ✅ **В 5× быстрее** с BLAKE3 vs MD5
- ✅ **Возобновляемое** сканирование (fdupes перезапускается с нуля)
- ✅ **Безопасная память** на больших датасетах (fdupes может упасть с OOM)
- ✅ **Лучший UX** с прогресс-барами и богатым выводом

### vs. rdfind

- ✅ **Параллельное хеширование** (rdfind однопоточный)
- ✅ **Современное хеширование** (BLAKE3 vs SHA-1)
- ✅ **Возобновляемое** сканирование
- ✅ **Лучшее тестирование** (100% vs ~60% покрытие)

### vs. jdupes

- ✅ **Больше стратегий сохранения** (5 vs 3)
- ✅ **Форматы экспорта** (JSONL, CSV, HTML)
- ✅ **Оптимизация для HDD** (сортировка inode, адаптивная параллельность)
- ✅ **Типобезопасность** (строгий mypy vs нетипизированный C)

## 🐛 Решение проблем

### Ошибки "Out of memory"

dupkiller использует константную память, но если всё ещё возникает OOM:

```bash
# Уменьшить количество воркеров
dupkiller scan /data --workers 4

# Ограничить максимальную пропускную способность
dupkiller scan /data --max-throughput 50MB
```

### Медленное сканирование на HDD

dupkiller автоматически определяет HDD и оптимизирует для них, но вы можете помочь:

```bash
# Уменьшить параллельность для HDD
dupkiller scan /mnt/hdd --workers 2

# Использовать ionice (только Linux)
ionice -c3 dupkiller scan /mnt/hdd
```

### Возобновление не работает

Проверьте, что используете **тот же** путь `--session`:

```bash
# ✅ Правильно
dupkiller scan /data --session /tmp/scan.db
dupkiller scan /data --session /tmp/scan.db  # тот же путь

# ❌ Неправильно (создаёт новую сессию)
dupkiller scan /data --session /tmp/scan.db
dupkiller scan /data --session /tmp/scan2.db  # другой путь
```
